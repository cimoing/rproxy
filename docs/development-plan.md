# RProxy 分阶段开发计划

## 1. 开发原则

RProxy 首批聚焦 Windows 平台，以 Rust 代理核心和 Slint GUI 为主线，先完成可运行、可配置、可分流的本地代理客户端，再逐步补齐 Tun、订阅兼容和更多协议能力。

阶段划分遵循以下原则：

- 先打通核心代理链路，再扩展透明代理能力。
- 先使用 YAML 本地配置，再支持订阅格式兼容。
- 自动路由、PAC 和 geosite 共享同一套路由决策模型。
- Windows 系统能力优先实现，包括系统代理、托盘和开机自启动。
- Tun 模式放入第二阶段，避免首版被权限、驱动和网络接口复杂度拖慢。

## 2. 阶段一：Windows MVP

### 2.1 阶段目标

完成 Windows 上可实际使用的 GUI 代理客户端，支持 YAML 配置、HTTP/SOCKS 本地监听、VLESS TLS + WebSocket 出站、自动路由、自动 PAC、系统代理、系统托盘和基础开机自启动配置。

### 2.2 功能范围

- Slint 主窗口。
- Rust 应用服务层。
- YAML 配置解析、校验和加载。
- 单配置管理。
- HTTP 本地代理监听。
- SOCKS 本地代理监听。
- VLESS 出站。
- VLESS TLS 支持。
- VLESS WebSocket 支持。
- 自动路由。
- 内置 geosite 数据库。
- 基于路由规则自动生成 PAC。
- 本地 PAC HTTP 服务。
- Windows 系统代理设置。
- Windows 系统托盘。
- 开机自启动开关。
- 启动、停止、重启代理。
- 基础运行日志。
- 错误提示和配置校验提示。

### 2.3 技术任务

- 建立 Rust workspace 与模块结构。
- 接入 Slint，完成主窗口和基础状态绑定。
- 定义 YAML 配置 schema。
- 实现配置文件加载、保存、校验和默认配置生成。
- 实现统一代理节点模型。
- 实现本地 HTTP 代理入口。
- 实现本地 SOCKS 代理入口。
- 实现 VLESS 出站适配器。
- 实现 TLS 连接能力。
- 实现 WebSocket 传输封装。
- 实现路由规则模型和匹配器。
- 将 geosite 数据库作为构建期资源打包。
- 实现 PAC 生成器和本地 PAC 服务。
- 实现 Windows 系统代理写入和恢复。
- 实现 Windows 托盘菜单。
- 实现 Windows 开机自启动配置。
- 接入日志系统。

### 2.4 验收标准

- 用户可以在 Windows 上启动 GUI。
- 用户可以加载 YAML 配置。
- 用户可以启动和停止代理。
- HTTP 与 SOCKS 本地端口可以正常连接。
- VLESS TLS + WebSocket 节点可以建立出站连接。
- 自动路由能根据自定义规则和 geosite 规则返回 Direct 或 Proxy。
- PAC 地址可被系统代理使用。
- 托盘可以显示运行状态，并提供启动、停止、退出入口。
- 开机自启动开关可以正确写入和取消。
- 代理停止后可以恢复系统代理状态。

### 2.5 不包含内容

- Tun 模式。
- 多配置管理完整 UI。
- 订阅格式兼容。
- geosite 在线更新。
- macOS 与 Linux 支持。
- 更多代理协议。

## 3. 阶段二：Tun 与多配置

### 3.1 阶段目标

在阶段一稳定运行的基础上，增加 Tun 模式和多配置管理，使 RProxy 可以处理不支持系统代理的应用流量，并支持用户在多个使用场景之间快速切换。

### 3.2 功能范围

- Windows Tun 模式。
- Tun 权限检查和启动引导。
- Tun 虚拟接口创建与释放。
- Tun 流量接入代理核心。
- Tun 与自动路由联动。
- 多配置管理。
- 配置创建、编辑、删除。
- 配置导入、导出。
- 默认配置设置。
- 配置切换时自动刷新代理、路由、PAC 和系统代理状态。
- 规则集管理基础能力。
- 更完整的运行日志和故障诊断。

### 3.3 技术任务

- 选型 Windows Tun 技术方案。
- 封装 Tun 平台适配层。
- 实现 Tun 生命周期管理。
- 将 Tun 流量接入路由决策模块。
- 建立多配置存储目录结构。
- 实现配置索引和当前活动配置记录。
- 实现配置热切换流程。
- 完善配置迁移机制。
- 完善权限错误、端口占用、接口创建失败等诊断。

### 3.4 验收标准

- 用户可以启用和关闭 Tun 模式。
- 不走系统代理的应用流量可被 Tun 接管。
- Tun 流量可以按路由规则直连或代理。
- 用户可以维护多个 YAML 配置。
- 用户可以在 GUI 中切换配置。
- 配置切换后 PAC、系统代理、监听端口和路由状态正确刷新。
- Tun 失败时能给出明确原因和恢复建议。

## 4. 阶段三：订阅兼容与扩展能力

### 4.1 阶段目标

增强配置来源和协议扩展能力，降低用户迁移成本，并提供更完整的日常使用体验。

### 4.2 功能范围

- 订阅格式兼容。
- Clash 配置导入。
- Sing-box 配置导入。
- 自定义订阅格式预留。
- 更多协议支持。
- 节点延迟测试。
- 节点健康检查。
- 流量统计。
- 配置备份与恢复。
- 更完整的日志过滤和导出。

### 4.3 技术任务

- 定义订阅导入中间模型。
- 实现 Clash 到 RProxy 配置转换。
- 实现 Sing-box 到 RProxy 配置转换。
- 扩展协议适配层。
- 实现节点探测和健康检查调度器。
- 实现流量统计采集。
- 完善 GUI 中的节点列表、检测状态和配置来源展示。

### 4.4 验收标准

- 用户可以导入 Clash 或 Sing-box 配置。
- 导入后的节点可以转换为 RProxy YAML 配置。
- 用户可以查看节点延迟和健康状态。
- 用户可以查看基础流量统计。
- 新协议可通过适配层接入，不影响现有 HTTP、SOCKS、VLESS 能力。

## 5. 阶段四：跨平台与长期演进

### 5.1 阶段目标

在 Windows 版本稳定后，逐步扩展到 macOS 和 Linux，并补齐更新、同步和更高级路由能力。

### 5.2 功能范围

- macOS 支持。
- Linux 支持。
- 跨平台系统代理适配。
- 跨平台托盘能力。
- 跨平台开机自启动。
- 自动更新。
- geosite 数据库更新机制。
- 配置同步。
- 更高级的路由规则编辑器。

### 5.3 技术任务

- 抽象平台能力接口。
- 实现 macOS 平台适配。
- 实现 Linux 平台适配。
- 评估跨平台安装包方案。
- 设计安全的更新机制。
- 设计 geosite 数据库版本管理。
- 设计配置同步和冲突处理策略。

### 5.4 验收标准

- macOS 和 Linux 可运行核心 GUI 与代理能力。
- 不同平台的系统代理、托盘和自启动行为一致。
- geosite 可以按版本更新。
- 配置同步不会破坏本地可用配置。

## 6. 阶段依赖

```text
阶段一 Windows MVP
  -> 阶段二 Tun 与多配置
    -> 阶段三 订阅兼容与扩展能力
      -> 阶段四 跨平台与长期演进
```

阶段二依赖阶段一的配置模型、代理核心、路由模块和 Windows 平台适配层。阶段三依赖阶段二稳定的多配置模型。阶段四依赖前三个阶段形成清晰的平台抽象和可复用核心能力。

## 7. 近期建议拆分

建议先按以下顺序启动开发：

1. Rust workspace 初始化。
2. Slint 空窗口和应用启动流程。
3. YAML 配置 schema 与默认配置。
4. HTTP/SOCKS 本地监听。
5. VLESS TLS + WebSocket 出站。
6. 路由模块与 geosite 打包。
7. PAC 生成和系统代理设置。
8. 托盘与开机自启动。
9. Windows 安装包。
